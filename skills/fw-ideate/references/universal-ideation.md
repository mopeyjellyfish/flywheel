# Universal Ideation

Load this file when the topic has no meaningful repo surface or no software
implementation surface.

This reference replaces the software-flavored framing from the main skill. The
core mechanism stays the same:

- ground the topic
- generate many candidates
- critique them adversarially
- present only the strongest survivors

Use the same tagged working shapes from the main skill:
`<focus_hint>`, `<grounding_summary>`, `<raw_candidates>`, and
`<survivor_shortlist>`.

## Role

Be a divergent thinking partner, not a delivery service.

Do not converge too early. The point of ideation is to surface stronger
candidates than the user would likely get from a quick first pass.

Match tone to the topic:

- business or strategy -> lead with tradeoffs and constraints
- creative work -> lead with range and contrast
- personal decisions -> lead with values and consequences

## How to Start

Match depth to scope:

- **Quick** - starter set now; one round, light critique, 3-5 survivors
- **Standard** - one or two clarifying questions, one round, 5-7 survivors
- **Full** - richer intake, multiple frames, deeper critique, 5-7 survivors

Use the discrimination test before asking anything: would changing one piece of
context materially change which ideas survive? If yes, proceed without
questions. If no, ask 1-3 narrowly chosen questions and stop once the answer
space is meaningfully constrained.

When the user has already provided rich context, confirm it briefly and move
on.

## Generation Frames

Generate the full candidate list before critiquing anything. Use the same six
frames from the main skill in domain-agnostic language:

1. **Pain and friction** - what is repeatedly annoying, slow, or unsatisfying?
2. **Inversion, removal, or automation** - what becomes possible if a step is
   removed, inverted, or delegated?
3. **Assumption-breaking and reframing** - what is being treated as fixed that
   is actually a choice?
4. **Leverage and compounding** - what creates second-order benefits?
5. **Cross-domain analogy** - how do unrelated fields solve a structurally
   similar problem?
6. **Constraint-flipping** - what becomes visible if the obvious constraint is
   pushed to an extreme or inverted?

Aim for 5-8 ideas per frame. Merge, dedupe, and keep only the strongest
cross-cutting combinations.

Follow the main skill's delegation gate. Do not delegate by default just
because the host supports it.

## Convergence

Reject weak candidates explicitly. For each rejected idea, keep a one-line
reason.

Score survivors using the same rubric from the main workflow:

- groundedness in the stated context
- expected value
- leverage
- pragmatism
- novelty
- implementation burden
- overlap with stronger ideas

Target 5-7 survivors by default. If fewer survive, report that honestly rather
than lowering the bar.

## Wrap-Up

Present survivors before any persistence. For each, include:

- title
- description
- rationale
- downsides
- confidence
- complexity

Then give a brief rejection summary so the user can see the cut line.

Read `references/shortlist-template.md` before presenting the final shortlist.

Persistence is still opt-in. Use the same save, brainstorm, refine, or no-save
options from `references/post-ideation-workflow.md`.
